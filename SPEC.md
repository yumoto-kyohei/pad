# Puzzle Practice 仕様書

パズドラ(パズル&ドラゴンズ)の盤面練習用ウェブアプリ。padsim をベースに改造したもの。
実装作業をAI(または人間)に依頼する際は、この仕様書を前提知識として渡すこと。

- 公開URL: https://pad-practice.github.io/pad/
- リポジトリ: https://github.com/pad-practice/pad (main ブランチ直プッシュ、GitHub Pages で配信)

## 1. 技術スタックと構成

ビルド工程なしの静的サイト。フレームワーク不使用、jQuery 1.11.3 + jQuery UI 1.11.3(CDN)+ touch-punch(タッチ対応)のみ。

| ファイル | 役割 |
|---|---|
| `index.html` | パズル練習ページ本体。UIはすべてここに静的記述、`onclick="requestAction(...)"` 方式 |
| `pad.js` | 練習ページの全ロジック(約2000行、グローバル変数ベース) |
| `pad.css` | 練習ページ+共通スタイル |
| `rankings.html` / `rankings.js` | 歴代王者ページ |
| `speedrun.html` / `speedrun.js` | 歴代最速ページ |
| `rankings.css` | 歴代王者・歴代最速の共通スタイル(両ページとも同じ id 構成で読み込む) |
| `sheetData.js` | 歴代王者・歴代最速で共通のスプレッドシート取得+CSVパース処理(`fetchSheetRecords`) |
| `menu.js` / `menu.css` | ハンバーガーメニュー(3ページ間の遷移) |
| `img/` | ドロップ画像(Red.png 等 8色+Fire.png 等の旧名エイリアス)+PWA用アイコン(下記) |
| `manifest.json` | PWAマニフェスト(3ページ共通で読み込む) |
| `sw.js` | PWA用 Service Worker。アプリシェルをキャッシュしオフライン動作させる |
| `pwa.js` | Service Worker登録スクリプト(3ページ共通) |

CSS/JSの読み込みには `?v5` のような手動キャッシュバスターを付けている。**CSSやJSを変更したら、それを読み込んでいる全ページのHTML側のクエリを上げること。**

### 1.1 PWA(ホーム画面追加・オフライン対応)

- 3ページすべての `<head>` に `<link rel="manifest" href="manifest.json">`、`theme-color`、iOS用の `apple-mobile-web-app-*` meta、`apple-touch-icon` を記載。`</body>` 直前で `pwa.js` を読み込み、`navigator.serviceWorker.register('sw.js')` を実行する。
- アイコンは `img/icon-192.png` / `img/icon-512.png` / `img/icon-maskable-512.png`(Android maskable用、セーフゾーンを空けた図案)/ `img/apple-touch-icon.png`(180×180、iOS用、不透明背景必須)の4種。現在は**仮アイコン**(ダークテーマの配色に合わせた紺グラデーション地に赤/青/緑のドロップ風の丸を3つ配置しただけの簡素なもの)。差し替える場合はこの4サイズ分を作り直すこと。生成に使った手順は `img/` 内に残していないので、次に作り直す際は同じ配色(`--panel-bg` 系のグラデーション)で作るとテーマに馴染む。
- `sw.js` はネットワークを介さず動く**パズル練習ページ(index.html)を完全オフライン対応**させることが主目的。`APP_SHELL` 配列にある全ファイル(HTML/CSS/JS/画像、ドロップ画像込み)をインストール時に個別キャッシュ(`cache.add` を1件ずつ、`addAll` ではない→1件失敗しても他は失敗しない)。fetchイベントではキャッシュ優先+バックグラウンド更新、ただし **`docs.google.com` へのリクエストは絶対にインターセプトしない**(歴代王者・最速チャレンジの生データを古いまま出さないため)。
- **重要**: `pad.css`/`pad.js` 等の `?vN` を上げたら、`sw.js` の `APP_SHELL` 配列内の該当エントリも同じバージョンに書き換え、かつ `CACHE_NAME` の末尾番号(`puzzle-practice-v1` など)をインクリメントすること。`CACHE_NAME` を上げないと、インストール済みのユーザーの端末で古いキャッシュが使われ続け、`activate` 時の古キャッシュ削除処理も走らない。
- ローカル確認方法: `cache.add` は個別に失敗を握りつぶすので、`caches.open('puzzle-practice-v1').then(c=>c.keys())` で実際に何件キャッシュされたかをブラウザのコンソールで確認できる(全35ファイルキャッシュされていることを確認済み)。

**全体はダークテーマ**(黒〜濃紺の背景+青/紫のアクセント)。配色は `pad.css` 冒頭の `:root` に集約している:
- `--accent`(青 #5b86ff)/`--accent2`(紫 #9a5cff)/`--accent-gradient`(この2色のグラデーション。アクティブなボタン・ナビ・見出し下線・ホバー塗り等に使用)
- `--panel-bg`(パネル地色・濃紺)/`--panel-bg-2`(ボタンや入力欄のやや明るい地色)/`--panel-hover`(ホバー時)/`--panel-border`(枠線)
- `--text`(明るい本文色。`body` に設定。**フォーム要素は color を継承しないので、テキストを持つ `button`/`input` には個別に `color:var(--text)` を指定する必要がある** — 黒文字が濃背景に乗る事故を防ぐため)/`--text-muted`(補助テキスト)/`--shadow`

`body` の背景は `radial-gradient(... #17213f → #05070d) fixed`。全ページとも `pad.css` を読み込むので `menu.css`/`rankings.css` からもこれらの変数を参照できる。新しく色を足すときは決め打ちの16進色を書かず、この変数を使うか変数を増やすこと(アクセントグラデーション上の白文字 `#fff` と、空マスの黒 `.black` は例外的に固定色)。

## 2. 盤面の内部表現(重要な前提)

- 盤面サイズは 5×6(デフォルト)/ 4×5 / 6×7 の3種。**変数名に注意: `rows` は横のマス数(=6)、`cols` は縦のマス数(=5)** という逆転した命名になっている(padsim由来)。
- マスの位置は一次元インデックス `position = y * rows + x`(x: 左から、y: 上から)。変換は `convertPosition(x,y)` / `convertXY(position, 'x'|'y')`。
- 各マスは `#tiles` 内の div。色は `tileColor` 属性(`red/blue/green/light/dark/heart/jammer/poison/black`)。`black` は空マス。
- 1マスの描画サイズは `scale`(基準90px)。`applyResponsiveLayout()` が画面幅に応じて縮小し、canvas `#arrowSurface` も盤面と同サイズに追従させる。
- 操作履歴は `replayMoveSet`: 通過したマスの position の配列(先頭=掴んだ位置)。

## 3. パズル練習ページの機能

上部トグルで3モードを切り替える(`appMode`: play / edit / analyze)。

### 3.1 Play モード

- ドラッグでドロップを移動。隣接マスとスワップし、`replayMoveSet` に記録。指を離すと `solveBoard()` がコンボ判定 → 消滅 → 落下(落ちコンONなら補充)を繰り返す。
- タイマー2種(`measureOn`): **制限**(カウントダウン、`timerTime` 初期4秒、+/-ボタンで増減)と **計測**(ストップウォッチ、離した時点の操作時間を表示)。
- 結果表示: コンボ数、手数、時間、盤面理論値(`maxPossibleCombosFromCounts`)、残ドロップ数など。
- **単発/継続**(`playStyle`): 単発=終了後に元の盤面へ自動リセット(1秒後、ルートは表示保持)。継続=消えた分をランダム補充して続行(`refillBoard`)。
- ボタン群: Reset(保存盤面に戻す)/ Replay(直前ルートを再生)/ Share(盤面文字列をテキストエリアへ)/ CtW(10秒間自由に動かせる、リプレイなし)/ 落ちコン(スカイフォールON/OFF)。
- 盤面文字列: 1マス=1文字(R,B,G,L,D,H,J,P,X)。`apply` で適用、`convert` で色置換(H→G など)。
- **CtW のタイマー**: `changeTheWorld()` はボタンを押した瞬間ではなく、`bindTileDragDrop()` の droppable `over` ハンドラで最初のスワップが起きた時に `startChangeTheWorldTimer()` を呼んで開始する(制限/計測モードの「タイマーは最初のスワップから」という既存の挙動と揃えてある)。10秒のカウントも同じタイミングで始まる。
- **落ちコン時の落下速度**: `dropField()` は毎パス `totalFallSpeed`(300ms)をそのパス数で割った時間だけ待つ。パス数は `calculateMaxFallPasses()` が計算するが、**落ちコンON時はこの関数の分岐が異なる**: 落ちコンOFF時は「浮いている色付きマスの下にある黒マス数」で計算するが(既存タイルの落下距離のみ考慮)、落ちコンON時はそのカラムの**黒マスの総数**で計算する(補充される新規ドロップも同じ回数だけ下に落ちる必要があるため)。この分岐が無いと、たとえば1カラムが丸ごと空という場合に旧ロジックは0パスと誤判定し(浮いている色付きタイルが無いため)、`totalFallSpeed` がパス数で割られずそのまま1パスの待ち時間になってしまい、実際に必要な複数パス分(本来の数倍)待たされて体感が遅くなっていた。

### 3.2 Edit モード

- パレット(8色)をタップ→盤面をタップ/なぞりで塗る。
- 盤面サイズ切替(5×6 / 4×5 / 6×7)。
- **Random生成**: 色チップのON/OFFと個数指定(空欄=残りを均等配分)で盤面生成(`generateEditBoard`)。
- **スクショ読込**: ゲームのスクリーンショットから盤面を復元(`importBoardFromImage`)。盤面が画像の全幅・下端付近にある前提で、複数の下マージン候補を試し、彩度が最も高くなるグリッド位置を採用。各マスは HSV の色相で分類。**現在選択中の盤面サイズで読み取る**ため、先にサイズを合わせる必要がある。

### 3.3 解析モード(最大コンボルート探索)

`runAnalysis()`。ビームサーチ:

- 状態 = (盤面, カーソル位置, 経路)。1手 = 隣接マスとのスワップ。直前のマスへの即戻りは禁止。
- 評価 = 落ちコンなし完全連鎖シミュレーション(`simulateCascade`)のコンボ数×1000 + 隣接同色ペア数(`countAdjacentPairs`、序盤の芽を残すタイブレーク)。
- 精度3段階 `ANALYSIS_LEVELS`: 速い(深さ25/ビーム2200)、標準(35/4200)、詳細(50/4500)。体感 約2-3秒/7秒/12秒。
- 理論値に到達したら `extraDepthAfterBest` 手だけ粘って打ち切り。`setTimeout(step, 0)` で1手ずつ刻んでUIを固めない。
- 結果: 最大コンボ帯とその1つ下の帯から重複除去して最大5ルート。各ルートに「ルート表示」「再生」ボタン。

### 3.4 My Boards(表示名「保存した盤面」)

localStorage(キー `puzzlePracticeBoards`)に盤面文字列+サイズを保存・読込・削除。ブラウザローカルのみ。**この節以降(保存した盤面・関連メッセージ・CtW/リセット失敗時などのdisplayOutputメッセージ)はすべて日本語**(以前は英語だったものを翻訳済み)。上部のモードボタン群(Reset/Replay/Share/CtW等)は英語のまま維持する方針。

### 3.5 オプション

Options はダイアログではなく `#infobooth` へのHTML出力。設定項目: 使用ドロップ色、生成時のコンボ成立許可、ルート表示ON/OFF、コンボ結果アイコン表示、シャッフル使用、最低消滅個数(3/4/5)。**設定は永続化されない(リロードで初期値に戻る)。**(旧「?」ヘルプボタンと `action=='help'` は未使用のため削除済み)
- モバイルでは盤面とオプション表示を同時にスクロールなしで置けないため、Options を開くと `#wrapwrap` に `options-open` クラスが付き、**盤面と保存盤面リストを隠してオプションが画面を占める**。オプション先頭の「✕ 閉じる」(`action=='closeoptions'`)またはモード切替(`setmode`)でクラスが外れ盤面が戻る(`closeoptions` は `applyResponsiveLayout(true)` を呼び直す — 盤面は display:none だった間に再計測が必要なため)。盤面が隠れている間はドラッグ操作ができない=コンボ結果メッセージが `#infobooth` を上書きする事故が起きないので、この方式で競合しない。デスクトップ(幅 > 600px)では `options-open` の隠し規則は効かず、従来通り盤面の下にオプションが出る。

### 3.6 モバイルレイアウト(スクロールなし・盤面下部固定)

`@media (max-width: 600px)` で、実機パズドラのように**盤面を画面下部(おおよそ下 6割〜最下端)に固定し、ページ自体はスクロールさせない**レイアウトにしている。実機同様、盤面はフル幅・アスペクト比 rows/cols(6:5)なので高さは幅の 5/6 ≈ 画面の約 37〜40%。

- `#wrapwrap` を `height:100dvh`(未対応ブラウザ用に `100vh` フォールバック)の flex 縦並びにし、`box-sizing:border-box` + `padding-top/bottom: env(safe-area-inset-*)` でノッチ/ステータスバーとホームインジケーターを避ける(`env()` を効かせるため viewport meta に `viewport-fit=cover` を付与。index.html のみ。rankings/speedrun は付けていない=safe-area padding を入れていないので付けると逆にノッチ下に潜る)。
- **`#board` は DOM 上 `#game` の外(直後の兄弟)に移動済み**。flex の `order` は兄弟間でしか効かないため、`#game` の子のままだと最下部へ回せない。デスクトップ(非flex)では DOM 順のまま `#game` の直後に描画されるので**従来の見た目と同一**(ヘッダー→パネル→盤面→ボタン→保存盤面)。
- モバイルの `order`: `#lineback`(0)→`#buttons`(1)→`#game`パネル(2)→`#library`(3, `flex:1` + `overflow-y:auto` で余白を埋め、保存盤面が多ければ**この枠内だけスクロール**)→`#infobooth`(4)→`#board`(5, 最下部)。`#infobooth` はモバイルでは背景透過・パディング小にして空メッセージ時に余計なカードが残らないようにしている(`options-open` 時のみパネル背景を復帰)。
- 盤面サイズ(`scale`)は `applyResponsiveLayout()` が `#board` の実測幅 / rows で決める。フル幅なのでモバイルでもタイルが枠にぴったり収まる(検証: 375px 幅で scale=55、盤面 335×275 内側にタイルが過不足なく収まりページはスクロール不可)。

## 4. ルート表示の仕様(現行実装)

操作後・Replay時・解析結果表示時に、通ったルートを `#arrowSurface`(canvas)に描画する。ここは過去に最も試行錯誤した箇所なので、変更時は下記の設計意図を必ず理解してから触ること。

### 4.1 描画パイプライン

```
replayMoveSet (position列)
  → movePathPoints()          マス中心のピクセル座標列
  → computeCornerTrims()      各中間地点の角丸め開始・終了点を算出
  → drawRoutePath()           進行度→色を計算しつつペインターズアルゴリズムで区間ごとに描画
      → routeColorAt(t)       区間ごとのコア色(青→赤のグラデーション)
      → drawSmoothSegment()   角丸め(quadraticCurveTo)+二重ストローク(黒フチ+着色コア)、1区間=1ストローク
```

### 4.2 進行度グラデーション方式(重複区間の見分け方)

- 線は常に**マス中心を通る**(オフセットなし)。以前は同じ辺を複数回通る区間を左右にずらす「レーン方式」だったが、ずらすと角や継ぎ目が滑らかに繋がらず(ステッチ挿入・妥協点などの特殊処理が積み重なっていた)、複雑なルートではかえって見づらかったため廃止した。
- 定数: `ROUTE_LINE_WIDTH = 7`(線全幅)、`ROUTE_EDGE_WIDTH = 2.25`(黒フチ幅)。
- `routeColorAt(t)`: ルート全体に対する進行度 `t`(0=始点〜1=終点)を `hsl(210°→0°, 85%, 65%→55%)` にマッピングし、青→赤へ滑らかに変化させる。**分母はルート全体の手数(`moveSet.length`)で固定**しており、Replay で `revealCount` を1手ずつ増やして再描画しても、既に描かれた区間の色は変化しない。
- 描画は**1区間=1回のストローク呼び出し**で、隣接区間の描画範囲は重ならない(`computeCornerTrims` で角の入口・出口点を共有座標として求め、区間 i は前区間が残した出口点から次区間の入口点まで直線+必要なら角のカーブを描く)。以前の実装は「前後1点分を重ねた窓」で描く方式だったが、これだと窓の重なり部分を後の区間の色で毎回上書きしてしまい、Replay で既に描画済みの区間の色が再生の進行とともに変わってしまう不具合があったため、重なりなしの方式に変更した。
- 前後関係(`drawRoutePath`): 交差検出はせず、区間を移動順に描き重ねる**ペインターズアルゴリズム**。後の通過が自然に上に乗る。グラデーションと組み合わせることで「上に乗っている暖色の線=後の通過」と一目で分かる。
- 見た目: 黒フチ(#1a1a1a、コントラスト確保のため固定)+着色コア(`routeColorAt` の結果)の二重ストローク、角は `scale*0.3` 半径で丸め、始点に緑丸「s」・終点に赤丸「g」。
- `drawRoutePath(ctx, moveSet, revealCount)` は **`revealCount` で途中までの部分描画に対応済み**(Replay の1手ずつ表示に使用)。

### 4.3 既知の課題

- 手数が非常に多いルートでは色の刻みが細かくなり、隣接する区間の色差が小さく見えることがある。
- グラデーションだけでは「今どこまで進んだか」を止めて確認する手段がない(§7.1 案Bの手数スライダーが未実装)。

## 5. ランキングダンジョンページ(rankings)・最速チャレンジページ(speedrun)

メニュー上の表示名は「ランキングダンジョン」「最速チャレンジ」だが、ファイル名・内部の関数/変数名は歴史的経緯で `rankings`/`歴代王者`・`speedrun`/`歴代最速` のままなので、コード中のコメント等で見かけても同じものを指す。

どちらも同じ Google スプレッドシート(「リンクを知っている全員: 閲覧者」で共有)を CSV export URL で fetch する。取得・パース処理は `sheetData.js` に共通化されており(`SHEET_ID`/`SHEET_GID` もここで一元管理)、`fetchSheetRecords(onSuccess, onError)` を呼ぶだけでよい。`rankings.js`・`speedrun.js` はどちらもこの関数を呼び、異なる列を読んでそれぞれのDOMへ描画するだけの薄いスクリプト。

両ページとも `rankings.css` を共有し、同じ id 構成(`#rankingsRoot`/`#leaderboardSection`/`#leaderboardList`/`#historySection`/`#historyTable`/`#rankingsStatus`)でマークアップしている。新しい列を集計するページを増やす場合もこのパターン(同じ id 構成のHTML + シートの別の列を読むだけの `*.js`)を踏襲すると `rankings.css` がそのまま使い回せる。

### 5.1 ランキングダンジョン(rankings.html)

- 期待するCSV列: `No`(開催回・数値)、`ダンジョン名`、`優勝者名`、`本人`(別垢を同一人物に束ねるための本名。空なら優勝者名を使用)、`身元不明`(TRUE で正体不明を表す)。同じ No に複数行=同率優勝。
- 表示:
  - **優勝回数ランキング**: `本人` 単位で集計、回数降順。同数の場合は**その回数へ最初に到達した人(最新優勝の No が小さい方)を上位**にする。同順位は同じ順位番号を共有し、次の異なる回数は同率人数分ランクが飛ぶ(例: 7位・7位・9位)。1〜3位は金・銀・銅の配色+メダル絵文字。最新優勝ダンジョン名を併記。
  - **歴代一覧**: No 降順の表。優勝者名の後ろに、2回目以降の優勝は「(n回目)」、初登場の表示名には「(m人目)」を付ける(別垢でも表示名が新規なら新王者としてカウント。別垢の初登場は「(n回目・m人目)」のように併記)。`身元不明` が TRUE の場合は名前の後ろではなく **人目/回目の数字の直後に「?」** を付ける(例: 「(5人目?)」)。優勝者なしの回は「王者なし」。No が空の行(最速チャレンジ専用の行、下記)はここでは無視される。

### 5.2 最速チャレンジ(speedrun.html)

- 期待するCSV列: `最速チャレンジ名`、`最速優勝者`。ランキングダンジョンと**同じシートの別の列**に追加する想定で、行はランキングダンジョン側の行と揃える必要はない(No 列は空のままでよい)。この2列が両方埋まっている行だけを1レコードとして拾う。
- No/日付に相当する列が無いため、**シート内の記載順(上から下)がそのまま時系列**という前提で扱う。実データで確認済み: **上が最古、下が最新**(例: 一番上の「新千手チャレンジ」が最古=1回目、一番下の「大樹チャレンジ」が最新=23回目)。新しい結果は一番下に追記していく運用。
- 表示:
  - **最速回数ランキング**: `最速優勝者` 単位で集計、回数降順。同数の場合はシート上での出現位置が早い(＝より古い時点でこの回数に到達した)方を上位にする。順位共有・金銀銅はランキングダンジョンと同じロジック。
  - **歴代最速一覧**: シートの並び順を逆転して表示(最新が上)。No は元の記載順(古い方から1,2,3...)で振り、表はその降順(最新が上)で並べる。ランキングダンジョンの歴代一覧の No 降順表示と見た目を揃えるための採番で、実データに書かれた通し番号ではない。優勝者名の後ろに、2回目以降は「(n回目)」、初登場の表示名には「(m人目)」を付ける(ランキングダンジョンと同じロジック、古い→新しいの順で走査して判定)。※最速側には `身元不明` 列が無いので「?」表記は付かない。

## 6. デプロイと運用

- `main` に push すると GitHub Pages が自動反映(数分)。
- 変更フロー: ローカルで動作確認 → commit → push(このリポジトリは確認済みの変更なら自動 push してよい運用)。

## 7. 改善候補(優先度順)

### 7.1 ルート重複区間の視認性

- **案A: 進行度グラデーション — 実装済み**
  線のコア色をルートの進行度で変化させる(青→赤)。レーンずらしは廃止し、線は常にマス中心を通る。詳細は §4.2。
- **案B: 手数スクラバー(推奨・小改修・未実装)**
  `drawRoutePath` の `revealCount` 引数は実装済みなので、ルート表示中に `<input type="range" min="2" max="手数">` を出し、input のたびに canvas をクリアして部分再描画するだけ。重複区間も「どこまで進んだか」を自分のペースで追える。解析結果の確認にも有効。
- **案C: レーンずらし — 廃止済み**
  かつては幾何学的に線を分離していたが、角や継ぎ目の滑らかな接続が難しく、複雑なルートではグラデーションのみの現行方式より見づらかった。再実装は検討しない。
- 補助案: 線分中点に小さな進行方向チェブロン(矢印)を間引いて描く。曲がりの少ない長いルートで有効だが、描き込み過多になりやすいのでオプション扱い。

### 7.2 設定の永続化

オプション(使用色、最低消滅個数、ルート表示、単発/継続、盤面サイズなど)を localStorage に保存し、リロード後も維持する。実装は軽いが体験改善が大きい。

### 7.3 解析の応答性

ビームサーチをメインスレッドの `setTimeout` 刻みから Web Worker に移すと、詳細(50手)でもUIが完全に滑らかになり、ビーム幅も広げられる。`simulateCascade` 一式は DOM 非依存の純関数なので移行しやすい。

### 7.4 コード面(低優先・機能追加のついでに少しずつ)

- `requestAction` が if 連鎖の巨大ディスパッチャ(本人コメントに `// CLEAN IT UP`)。アクション名→関数のテーブルにするだけで見通しが良くなる。
- `rows`/`cols` の逆転命名は誤実装の温床。リネームは影響範囲が広いので、せめて本仕様書と関数コメントで注意喚起を維持する。
- jQuery 1.11.3(2015年)への依存はドラッグ&ドロップ(jQuery UI draggable/droppable)のみが本質。将来 Pointer Events への置き換えを検討。
- `rankings.js` 冒頭の「TODO: fill in once the Google Sheet is created」コメントは対応済みなので削除してよい。
