# Puzzle Practice 仕様書

パズドラ(パズル&ドラゴンズ)の盤面練習用ウェブアプリ。padsim をベースに改造したもの。
実装作業をAI(または人間)に依頼する際は、この仕様書を前提知識として渡すこと。

- 公開URL: https://pad-practice.github.io/pad/
- リポジトリ: https://github.com/pad-practice/pad (main ブランチ直プッシュ、GitHub Pages で配信)

## 1. 技術スタックと構成

ビルド工程なしの静的サイト。フレームワーク不使用、jQuery 1.11.3 + jQuery UI 1.11.3(CDN)+ touch-punch(タッチ対応)のみ。

| ファイル | 役割 |
|---|---|
| `index.html` | パズル練習ページ本体。UIはすべてここに静的記述、`onclick="requestAction(...)"` 方式 |
| `pad.js` | 練習ページの全ロジック(約2000行、グローバル変数ベース) |
| `pad.css` | 練習ページ+共通スタイル |
| `rankings.html` / `rankings.js` / `rankings.css` | 歴代王者ページ |
| `menu.js` / `menu.css` | ハンバーガーメニュー(2ページ間の遷移) |
| `img/` | ドロップ画像(Red.png 等 8色+Fire.png 等の旧名エイリアス) |

CSS/JSの読み込みには `?v5` のような手動キャッシュバスターを付けている。**CSSやJSを変更したら index.html / rankings.html 側のクエリを上げること。**

## 2. 盤面の内部表現(重要な前提)

- 盤面サイズは 5×6(デフォルト)/ 4×5 / 6×7 の3種。**変数名に注意: `rows` は横のマス数(=6)、`cols` は縦のマス数(=5)** という逆転した命名になっている(padsim由来)。
- マスの位置は一次元インデックス `position = y * rows + x`(x: 左から、y: 上から)。変換は `convertPosition(x,y)` / `convertXY(position, 'x'|'y')`。
- 各マスは `#tiles` 内の div。色は `tileColor` 属性(`red/blue/green/light/dark/heart/jammer/poison/black`)。`black` は空マス。
- 1マスの描画サイズは `scale`(基準90px)。`applyResponsiveLayout()` が画面幅に応じて縮小し、canvas `#arrowSurface` も盤面と同サイズに追従させる。
- 操作履歴は `replayMoveSet`: 通過したマスの position の配列(先頭=掴んだ位置)。

## 3. パズル練習ページの機能

上部トグルで3モードを切り替える(`appMode`: play / edit / analyze)。

### 3.1 Play モード

- ドラッグでドロップを移動。隣接マスとスワップし、`replayMoveSet` に記録。指を離すと `solveBoard()` がコンボ判定 → 消滅 → 落下(落ちコンONなら補充)を繰り返す。
- タイマー2種(`measureOn`): **制限**(カウントダウン、`timerTime` 初期4秒、+/-ボタンで増減)と **計測**(ストップウォッチ、離した時点の操作時間を表示)。
- 結果表示: コンボ数、手数、時間、盤面理論値(`maxPossibleCombosFromCounts`)、残ドロップ数など。
- **単発/継続**(`playStyle`): 単発=終了後に元の盤面へ自動リセット(1秒後、ルートは表示保持)。継続=消えた分をランダム補充して続行(`refillBoard`)。
- ボタン群: Reset(保存盤面に戻す)/ Replay(直前ルートを再生)/ Share(盤面文字列をテキストエリアへ)/ CtW(10秒間自由に動かせる、リプレイなし)/ 落ちコン(スカイフォールON/OFF)。
- 盤面文字列: 1マス=1文字(R,B,G,L,D,H,J,P,X)。`apply` で適用、`convert` で色置換(H→G など)。

### 3.2 Edit モード

- パレット(8色)をタップ→盤面をタップ/なぞりで塗る。
- 盤面サイズ切替(5×6 / 4×5 / 6×7)。
- **Random生成**: 色チップのON/OFFと個数指定(空欄=残りを均等配分)で盤面生成(`generateEditBoard`)。
- **スクショ読込**: ゲームのスクリーンショットから盤面を復元(`importBoardFromImage`)。盤面が画像の全幅・下端付近にある前提で、複数の下マージン候補を試し、彩度が最も高くなるグリッド位置を採用。各マスは HSV の色相で分類。**現在選択中の盤面サイズで読み取る**ため、先にサイズを合わせる必要がある。

### 3.3 解析モード(最大コンボルート探索)

`runAnalysis()`。ビームサーチ:

- 状態 = (盤面, カーソル位置, 経路)。1手 = 隣接マスとのスワップ。直前のマスへの即戻りは禁止。
- 評価 = 落ちコンなし完全連鎖シミュレーション(`simulateCascade`)のコンボ数×1000 + 隣接同色ペア数(`countAdjacentPairs`、序盤の芽を残すタイブレーク)。
- 精度3段階 `ANALYSIS_LEVELS`: 速い(深さ25/ビーム2200)、標準(35/4200)、詳細(50/4500)。体感 約2-3秒/7秒/12秒。
- 理論値に到達したら `extraDepthAfterBest` 手だけ粘って打ち切り。`setTimeout(step, 0)` で1手ずつ刻んでUIを固めない。
- 結果: 最大コンボ帯とその1つ下の帯から重複除去して最大5ルート。各ルートに「ルート表示」「再生」ボタン。

### 3.4 My Boards

localStorage(キー `puzzlePracticeBoards`)に盤面文字列+サイズを保存・読込・削除。ブラウザローカルのみ。

### 3.5 オプション

`?`(ヘルプ)と Options はダイアログではなく `#infobooth` へのHTML出力。設定項目: 使用ドロップ色、生成時のコンボ成立許可、ルート表示ON/OFF、コンボ結果アイコン表示、シャッフル使用、最低消滅個数(3/4/5)。**設定は永続化されない(リロードで初期値に戻る)。**

## 4. ルート表示の仕様(現行実装)

操作後・Replay時・解析結果表示時に、通ったルートを `#arrowSurface`(canvas)に描画する。ここは過去に最も試行錯誤した箇所なので、変更時は下記の設計意図を必ず理解してから触ること。

### 4.1 描画パイプライン

```
replayMoveSet (position列)
  → movePathPoints()          マス中心のピクセル座標列
  → computeSegmentOffsets()   線分ごとのレーンオフセット決定
  → buildOffsetPolyline()     オフセット適用済みの連続ポリライン生成
  → drawRoutePath()           ペインターズアルゴリズムで区間ごとに描画
      → drawSmoothPath()      角丸め(quadraticCurveTo)+二重ストローク
```

### 4.2 レーン方式(重複区間のずらし)

- 定数: `ROUTE_LINE_WIDTH = 7`(線全幅)、`ROUTE_EDGE_WIDTH = 2.25`(黒フチ幅)。隣接レーンの間隔はちょうど線幅1本分で、黒フチ同士が接して「継ぎ目」になる(隙間を空けない)。
- レーン割当(`computeSegmentOffsets`): 辺(マス間の接続)ごとに使用済みレーンを記録。**未使用の辺では現在のレーンを維持**し(中心に戻らない)、同じ辺を再通行するときだけ空きレーン(0, +1, -1, +2, -2 の順 = `laneAt`)へ移る。辺の向きは正規化(小さいindex→大きいindex)し、往復どちらでも同じレーン幾何になる。
  - ※「使い終わったら中央レーンに戻す」実装は過去に試して不自然な斜めのねじれが出たため廃止済み。
- レーン移行点の接続(`buildOffsetPolyline`):
  - 直進継続中の移行 → 旧レーン点と新レーン点を近接して2点挿入し、角丸めで短い「ステッチ」にする。
  - 曲がり角での移行 → 2点挿入すると角の丸めと干渉して折れて見えるため、**両オフセットの中間の1点**に妥協させる。
- 前後関係(`drawRoutePath`): 交差検出はせず、点列を移動順に小さな窓(前後1点の重なり付き)で描き重ねる**ペインターズアルゴリズム**。後の通過が自然に上に乗る。窓が重なるので同一レーンの連続は切れ目なく見える。
- 見た目: 黒フチ(#1a1a1a)+白コア(#fff)の二重ストローク、角は `scale*0.3` 半径で丸め、始点に緑丸「s」・終点に赤丸「g」。
- `drawRoutePath(ctx, moveSet, revealCount)` は **`revealCount` で途中までの部分描画に対応済み**(Replay の1手ずつ表示に使用)。

### 4.3 既知の課題

同じ辺を何度も通るルートでは、レーンずらしをしても視認性が十分でない:

- ずらすと線同士の滑らかな接続が難しく(上記の妥協点の積み重ね)、複雑なルートでは重ねて描くより読みにくくなることがある。
- どちらの線が「先」なのか、線を目で追わないと分からない。

改善案は §7.1 を参照。

## 5. 歴代王者ページ(rankings)

- データソース: Google スプレッドシート(「リンクを知っている全員: 閲覧者」で共有)を CSV export URL で fetch。`rankings.js` 冒頭の `SHEET_ID` / `GID` で指定。
- 期待するCSV列: `No`(開催回・数値)、`ダンジョン名`、`優勝者名`、`本人`(別垢を同一人物に束ねるための本名。空なら優勝者名を使用)、`身元不明`(TRUE で名前に「?」を付ける)。同じ No に複数行=同率優勝。
- 表示:
  - **優勝回数ランキング**: `本人` 単位で集計、回数降順。最新優勝ダンジョン名を併記。
  - **歴代一覧**: No 降順の表。優勝者名の後ろに「(n)」=その人のn回目の優勝、初登場の表示名には「(m人目の王者)」を付ける(別垢でも表示名が新規なら新王者としてカウント)。優勝者なしの回は「王者なし」。
- データ更新はスプレッドシート編集のみで完結し、デプロイ不要。

## 7. 改善候補(優先度順)

### 7.1 ルート重複区間の視認性(最優先の課題)

方針: **レーンずらし(幾何学的分離)だけで解決しようとしない。** 「どの線が後か」は色と時間で伝える方が滑らかさを犠牲にしない。

- **案A: 進行度グラデーション(推奨・小改修)**
  白コアの色をルートの進行度で変化させる(例: 序盤=水色 → 終盤=オレンジ〜赤の色相ランプ)。`drawRoutePath` は既に区間ごとに `drawSmoothPath` を呼んでいるため、区間indexから補間した色を `strokeStyle` に渡すだけでよい。黒フチは維持(盤面ドロップとのコントラスト確保)。ペインターズ順と組み合わせると「上に乗っている線=より暖色=後の通過」と一目で分かる。明度も同時に変えると色覚多様性にも有利。
- **案B: 手数スクラバー(推奨・小改修)**
  `drawRoutePath` の `revealCount` 引数は実装済みなので、ルート表示中に `<input type="range" min="2" max="手数">` を出し、input のたびに canvas をクリアして部分再描画するだけ。重複区間も「どこまで進んだか」を自分のペースで追える。解析結果の確認にも有効。
- **案C: レーンずらしの扱い**
  案A+Bを入れた上で、レーンずらしは現状維持(グラデーションと併用で「分離+順序」の両方が得られる)。それでも複雑なルートで見づらい場合のみ、`computeSegmentOffsets` を常に lane 0 を返すよう切り替えるオプション(中心線のみ+グラデーション)を追加して比較する。ずらし幅の調整や新しい接続方法の再発明はコストに対して効果が薄い(実績あり)。
- 補助案: 線分中点に小さな進行方向チェブロン(矢印)を間引いて描く。曲がりの少ない長いルートで有効だが、描き込み過多になりやすいのでオプション扱い。

### 7.2 設定の永続化

オプション(使用色、最低消滅個数、ルート表示、単発/継続、盤面サイズなど)を localStorage に保存し、リロード後も維持する。実装は軽いが体験改善が大きい。

### 7.3 解析の応答性

ビームサーチをメインスレッドの `setTimeout` 刻みから Web Worker に移すと、詳細(50手)でもUIが完全に滑らかになり、ビーム幅も広げられる。`simulateCascade` 一式は DOM 非依存の純関数なので移行しやすい。

### 7.4 コード面(低優先・機能追加のついでに少しずつ)

- `requestAction` が if 連鎖の巨大ディスパッチャ(本人コメントに `// CLEAN IT UP`)。アクション名→関数のテーブルにするだけで見通しが良くなる。
- `rows`/`cols` の逆転命名は誤実装の温床。リネームは影響範囲が広いので、せめて本仕様書と関数コメントで注意喚起を維持する。
- jQuery 1.11.3(2015年)への依存はドラッグ&ドロップ(jQuery UI draggable/droppable)のみが本質。将来 Pointer Events への置き換えを検討。
- `rankings.js` 冒頭の「TODO: fill in once the Google Sheet is created」コメントは対応済みなので削除してよい。
